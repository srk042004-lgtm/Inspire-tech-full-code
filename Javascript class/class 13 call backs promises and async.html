JavaScript's core is synchronous, meaning code runs line-by-line. To handle operations that take an unpredictable amount of time‚Äîlike network requests, file reading, or timers‚Äîwe need **Asynchronous Programming**. This section traces the evolution of handling async code in JavaScript.

PART 1: Callbacks (The Traditional Approach)

A **callback function** is simply a function passed as an argument to another function, which is then executed once the original function completes its task. Callbacks were the original mechanism for handling asynchronous results.

A. The `setTimeout` Example

The built-in `setTimeout` function is the simplest example of async execution using a callback. 

console.log("1. Starting operation...");

// This runs *after* the specified time (1000ms)
setTimeout(() => {
    console.log("3. Callback executed after 1 second."); 
}, 1000);

console.log("2. Synchronous code continues immediately.");
// Output Order: 1, 2, 3
B. The Problem: Callback Hell

When multiple asynchronous operations depend on each other, nesting callbacks leads to deeply indented, unreadable, and hard-to-maintain code, known as **"Callback Hell."** 

getUser(id, (user) => {
    getOrders(user.id, (orders) => {
        getDetails(orders[0].id, (details) => {
            // Very deep nesting, hard to track errors...
            console.log(details); 
        });
    });
});

PART 2: Promises (The Intermediate Solution)

A **Promise** is a cleaner, more readable structure used to handle the result of an asynchronous operation, serving as a placeholder for a value that is currently unknown.

A. Promise States
StateMeaningAction
**Pending**Initial state; operation is running.Waiting...
**Fulfilled**Operation completed successfully.Handled by **`.then()`**
**Rejected**Operation failed (an error occurred).Handled by **`.catch()`**
B. Chaining Promises

Promises allow us to chain `.then()` calls sequentially, which eliminates callback hell. The return value of one `.then()` block becomes the input for the next.

Using Fetch (A Promise-based API)
fetch('https://api.example.com/user/1') // Returns Promise 1 (HTTP Response)
    .then(response => {
        // First .then: Check response status & extract JSON (Returns Promise 2)
        if (!response.ok) throw new Error('Network error');
        return response.json(); 
    })
    .then(userData => {
        // Second .then: Work with the parsed data
        console.log("User data:", userData); 
    })
    .catch(error => {
        // Handle ANY error in the chain (Promise rejection or thrown error)
        console.error("Operation Failed:", error); 
    });

PART 3: Async / Await (The Modern Standard)

`async/await` is modern JavaScript syntax (ES2017+) built on top of Promises that makes asynchronous code look and behave like synchronous code, greatly improving readability.

A. The `async` Keyword

You must use `async` before a function declaration (`async function`) to signal that the function will execute asynchronous code. An `async` function always implicitly returns a **Promise**.

B. The `await` Keyword

The `await` keyword can **only** be used inside an `async` function. It pauses the execution of the `async` function until the Promise it is waiting for is **resolved** (fulfilled or rejected).

Example: Fetch using `async/await`
async function loadUserData() {
    try {
        // 1. Await pauses until the HTTP response is received
        const response = await fetch('https://api.example.com/user/1'); 

        if (!response.ok) {
            throw new Error(`HTTP error! Status: ${response.status}`);
        }

        // 2. Await pauses until the JSON data is parsed
        const userData = await response.json(); 
        
        console.log("User data loaded:", userData);
        return userData;

    } catch (error) {
        // 3. Errors are handled cleanly using try...catch
        console.error("Could not fetch user data:", error.message);
    }
}

// Call the async function
loadUserData();
**Key Takeaway:** While Callbacks were the first step, **Promises** provide the underlying structure, and **`async/await`** provides the best, cleanest syntax for working with that structure today.
